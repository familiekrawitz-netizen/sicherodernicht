2. Beispiel-Firma: safeguard1
=============================

In der App gibt es bereits einen Firmenzugang mit dem Beispielnamen:

safeguard1

Dieser Zugang steht fuer eine Firma bzw. ein Unternehmen. Mitarbeiter dieser Firma koennen als Firmennutzer angelegt und dieser Firma zugeordnet werden.

Firmennutzer haben mehr Rechte als private Nutzer. Sie koennen zum Beispiel:

- die Meldung "6 Aktuelle Gefahr" abgeben
- die Firmen-Werkzeuge bzw. den Einsatzmodus nutzen
- Teamstandorte anzeigen, sofern Mitarbeiter ihren Standort teilen
- besondere Gefahrenmeldungen wie Schusswaffe, Messer, Gewalt oder Mob melden
- aktuelle Gefahren der letzten 24 Stunden im Firmenmodus sehen

4. Berechtigungen wieder entziehen
==================================

Wenn ein Mitarbeiter keinen Zugang mehr haben soll, gibt es zwei einfache Wege:

1. Nutzer auf der Adminseite loeschen.
Das ist die klarste Loesung, wenn der Zugang dauerhaft entzogen werden soll.

2. PIN oder Login-Code aendern.
Das ist sinnvoll, wenn der Nutzer spaeter vielleicht wieder Zugriff erhalten soll.

Wichtig:
Wenn ein Mitarbeiter ausscheidet, sollte sein Login sofort geloescht oder geaendert werden.


5. Welche Daten werden gespeichert?
===================================

Oeffentliche Bewertungen 1 bis 5:
Diese werden anonym gespeichert. Sie dienen dem langfristigen Lagebild.

Fun-Meldungen:
Diese werden ebenfalls anonym gespeichert.

Meldung "6 Aktuelle Gefahr":
Die eigentliche Meldung wird anonym im Lagebild gespeichert.
Zusaetzlich gibt es fuer registrierte Nutzer eine getrennte Audit-Speicherung, damit Missbrauch geprueft werden kann.

Firmennutzer:
Bei Firmennutzern werden sicherheitsrelevante Meldungen nachvollziehbar gespeichert, zum Beispiel wer, wann, wo und was gemeldet hat. Diese Audit-Daten sind nur fuer Adminzwecke gedacht.


6. Datenbank lokal auf dem Mac sichern
======================================

Die lokale Datenbankdatei liegt hier:

/Users/peterkrawitz/Documents/Web-App/data/store.json

Zum Sichern kannst du im Finder diesen Ordner oeffnen:

/Users/peterkrawitz/Documents/Web-App/data

Dann kopierst du die Datei:

store.json

an einen sicheren Ort, zum Beispiel auf einen USB-Stick oder in einen Sicherungsordner.

Wichtig:
Diese Datei kann sensible Daten enthalten, insbesondere Admin-, Nutzer- oder Audit-Daten. Sie sollte nicht unverschluesselt an Unbefugte weitergegeben werden.


7. Datenbank von Render auf den Laptop uebertragen
==================================================

Das genaue Vorgehen haengt davon ab, wie die App bei Render betrieben wird.

Fall A: Render nutzt noch die einfache Datei data/store.json
-----------------------------------------------------------

Dann muss die Datei von Render heruntergeladen werden.

Je nach Render-Plan und Einrichtung geht das zum Beispiel ueber:

- Render Shell
- ein Admin-Export in der App
- spaeter ueber eine geschuetzte Exportfunktion

Der aktuell einfachste Weg in unserer App ist:

1. Adminseite der Online-App oeffnen.
2. Als Admin einloggen.
3. Im Bereich "Datenbank & Export" den Export nutzen.
4. Fuer das langfristige Lagebild: "Alles als CSV exportieren" oder JSON.
5. Fuer nachvollziehbare Admin-/Auditdaten: "Audit-Daten CSV" oder JSON.
6. Die heruntergeladene Datei auf dem Laptop speichern.

Der normale Lagebild-Export enthaelt keine Meldernamen.
Der Audit-Export enthaelt nachvollziehbare Daten und muss besonders geschuetzt werden.


Fall B: Render nutzt spaeter eine echte Datenbank
-------------------------------------------------

Fuer den echten Betrieb ist eine richtige Datenbank besser, zum Beispiel:

- PostgreSQL bei Render
- Supabase
- Neon

Dann erfolgt der Transfer nicht mehr ueber die Datei store.json, sondern ueber einen Datenbankexport.

Typische Moeglichkeiten:

- Export als CSV ueber die Adminseite
- Export als JSON ueber die Adminseite
- Datenbank-Dump direkt beim Anbieter

Fuer dich als Betreiber waere die einfachste Variante:
Die App bekommt eine Adminseite, auf der du per Button die gewuenschten Daten herunterladen kannst.


8. Empfehlung fuer den Echtbetrieb
==================================

Fuer Tests ist data/store.json in Ordnung.

Fuer den echten oeffentlichen Betrieb sollte spaeter Folgendes umgesetzt werden:

- echte Datenbank statt einfacher JSON-Datei
- regelmaessige automatische Backups
- Adminzugang nicht im Code speichern
- Audit-Daten getrennt halten
- Audit-Daten zeitlich begrenzen
- Lagebilddaten dauerhaft, aber anonym speichern
- Export nur fuer Admins erlauben
- Datenschutztext an den echten Betrieb anpassen
